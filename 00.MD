# PROJECT_KNOWLEDGE.md

## 1. Project Vision
This project is a "Multi-user Real-time Generative Art Platform" designed for educational environments. It is not a standard collaborative whiteboard. It is a classroom-based system where dozens of students submit digital assets (physical drawings captured via camera), and a central teacher client uses WebGL/generative algorithms to dynamically render these assets into a massive, interactive mosaic based on a reference image or live webcam feed.

## 2. Core Architecture: Asymmetric Client Model
DO NOT implement standard CRDT-based multi-user canvas syncing (like Yjs) for the main canvas, as it will crash with 50-60 concurrent 3D rendering updates. Instead, use an **Asymmetric Architecture**:

* **Student Client (Lightweight Contributor):** A mobile-friendly or web interface acting as an "Asset Collector." Students capture/upload materials, perform client-side preprocessing (cropping), and submit these asset packages to the server. They DO NOT render the heavy 3D scene. They receive a broadcasted image/stream of the final result.
* **Teacher Client (Heavy Master Renderer):** A desktop-class web app leveraging local GPU. It connects to a WebSocket/Realtime channel to receive student submissions into a "Staging Queue." The teacher selects assets from the queue to push into the main rendering engine.
* **Server/Database:** Acts as the Auth provider, room state manager, real-time message broker, and persistent asset storage.

## 3. Core Feature: "The 10-Level Grayscale Brushstroke Mosaic"
This specific activity defines the primary user flow and technical requirements:

### Student Flow (The Asset Collector)
1.  **Task:** The student is tasked to draw 10 square brushstrokes using physical pens/pencils on paper, representing a luminance gradient from 1 (darkest) to 10 (lightest). They can use various tools (blue ballpoint pens, graphite pencils, etc.).
2.  **UI Guidance:** The app displays 10 reference gradient squares. 
3.  **Capture & Crop:** The student uses the device camera to capture their drawings. The frontend guides them to crop each capture into a 100x100 pixel square. **Crucial:** DO NOT apply any grayscale or desaturation filters. Preserve the original RGB colors (e.g., the blue ink color) captured by the camera.
4.  **Refinement:** Students can preview the 10 slots and retake any unsatisfactory captures before submission.
5.  **Submission:** Submits the 10-image payload (a "material package") tagged with their Student ID to the server.

### Teacher Flow (The Master Renderer)
1.  **Input Source:** The teacher selects a static reference image or grants webcam access for a live video feed.
2.  **Pre-processing:** The input source is scaled so the longest edge is 1500 pixels. The logic divides the image into a grid of 10x10 pixel blocks.
3.  **Real-time Analysis:** Calculates the average brightness/luminance of each 10x10 block and maps it to a 1-10 integer scale.
4.  **Material Queue:** The teacher UI displays a real-time incoming list of student material packages.
5.  **The Render Engine (Crucial for Performance):** When the teacher clicks a student's package, the canvas renders the mosaic. Since replacing 15,000 blocks with 100x100 images results in a massive 150-Megapixel canvas, standard DOM/Canvas2D will crash. 
    * **Architecture Mandate:** Use WebGL (via Three.js or raw GLSL shaders). 
    * **Technique:** Compile the 10 student images into a single `1000x100` Texture Atlas. Pass the reference image/webcam feed as a base texture to a custom Fragment Shader. The shader calculates the luminance (grayscale value) of the base texture to determine the index (1-10). It then dynamically samples the corresponding 1/10th slice from the student's Texture Atlas. **Crucial:** The shader must output the original RGB color from the student's texture, NOT a desaturated version. This creates a rich, multi-colored mosaic built on luminance mapping, running at a smooth 60fps.

## 4. Tech Stack Requirements
* **Frontend Framework:** Next.js (App Router) + React + TypeScript + Tailwind CSS.
* **Camera/Image Processing:** MediaDevices API for camera, Canvas API for client-side cropping to 100x100.
* **Render Engine (Modular):** `Three.js` (with `@react-three/fiber`) specifically utilizing custom ShaderMaterials and Texture Atlas techniques for high-performance mosaic rendering. Keep it decoupled so future engines (like p5.js wrappers) can be added.
* **Backend & Database:** Supabase (PostgreSQL, Supabase Auth, Supabase Storage for the 100x100 images, and Supabase Realtime for updating the teacher's queue).

## 5. Roles & Permissions (RBAC)
* **Super Admin:** System-wide management.
* **Teacher (Room Owner):** Can create rooms, generate entry codes, manage the student queue, control the main canvas rendering, and save specific student submissions to their personal "Digital Asset Library" for future use.
* **Student (Editor/Submitter):** Enters rooms via a PIN code. Can only submit assets and view the broadcasted result. Cannot affect other students' submissions.

## 6. Core Data Entities (Supabase Schema Guidelines)
* `users`: Extended user profiles and roles.
* `rooms`: `id`, `teacher_id`, `join_code`, `status` (open/locked/rendering), `current_canvas_state` (JSONB).
* `room_participants`: Maps users to rooms.
* `assets`: **CRITICAL TABLE.** Must be highly flexible. Contains `id`, `owner_id`, `asset_type` (e.g., '10_level_brushstrokes'), `student_id` (who created it originally), and `metadata` (JSONB). For this task, the metadata must contain an array of 10 storage URLs ordered by luminance level (1-10).

## 7. Execution Guidelines for AI Assistant (Claude Code)
When receiving instructions from the user, adhere to these phased persona constraints:
1.  **Phase 1 (The Architect):** Always start by defining PostgreSQL schemas and Row Level Security (RLS) in Supabase. Design how the "10-image material package" is efficiently stored. Do not write frontend UI code until data models are secure.
2.  **Phase 2 (The Auth Guard):** Implement robust Next.js middleware, login flows, and room PIN verification. Protect room routes.
3.  **Phase 3 (The Student UI):** Focus on the MediaDevices camera capture, 10-slot state management, and 100x100 cropping UI before handling backend uploads.
4.  **Phase 4 (The WebGL Engine):** Do NOT use standard React `<img>` tags or 2D canvas loops for the teacher's mosaic. Implement the Three.js Fragment Shader logic using a Texture Atlas to ensure the 10x scale-up rendering is performant.

**Golden Rule:** Strictly type all Supabase database responses and generic JSONB metadata using TypeScript interfaces before implementing frontend state.

## 8. Modular Implementation (Agent Skills)
To ensure scalability, the AI should approach the project through these decoupled modules:

1. **Database & Security Skill:** Initialize `supabase/migrations`. Focus on RLS policies where `student` can only `INSERT` to `assets` and `SELECT` their own, while `teacher` can `SELECT` all assets in their `room_id`.
2. **Image Processing Skill:** Create a utility to merge 10 `100x100` canvas elements into a single `1000x100` DataURL/Blob for optimized storage and GPU texture binding.
3. **GLSL Shader Skill:** Implement a `CustomShaderMaterial` that:
   - Takes `uReferenceTexture` (the webcam/image).
   - Takes `uStudentTexture` (the 10-step atlas).
   - Logic: `float gray = dot(texColor.rgb, vec3(0.299, 0.587, 0.114));`
   - Map `gray` to `1/10` increments to shift `uv.x` on `uStudentTexture`.
4. **Realtime Sync Skill:** Use Supabase Realtime broadcast to notify the teacher's "Staging Queue" without triggering a full page re-render.
